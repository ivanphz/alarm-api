# 执行器通道总册 —— CHANNELS（alarm-api/docs/）

> **这份文档是什么**：iPhone 执行层的全部"打断/提醒/触发"能力清单与选型判据。
> 以后任何"我想被提醒/被打断/被触发"的新需求，**先查此表再设计**。
> 全部关键断言均有 2026-07-18 实测支撑（§6），实测优先于任何官方文档与直觉。

---

## 1. 输出通道谱系（按打断力降序）

| # | 通道 | 穿静音 | 穿专注 | 无人值守建 | 无人值守改 | 无人值守删 | 用户状态 | 备注 |
|---|---|---|---|---|---|---|---|---|
| 1 | Fixed 闹钟 `Gate-Fixed-*` | ✅ | ✅ | 预建 | 只开/关 | — | 无 | 自定义铃震；常驻最可靠 |
| 2 | 动态闹钟 `Gate-ES-*` 等 | ✅ | ✅ | ✅ | ❌（关旧建新） | 需人工确认 | 无 | 默认铃；漏同步=漏响 |
| 3 | Urgent 提醒（26.2） | ✅实测 | ✅实测 | ✅（Add 有 Urgent 参数） | ✅（铃伴生于日期，改期即改铃；Urgent 开关本身不可改） | 需人工确认 | 完成/贪睡 | 真闹钟+完成语义；只在 Find My 主设备；需"闹钟"权限 |
| 4 | Bark critical 推送 | ✅实测 | ✅实测 | ✅（服务端 POST） | — | — | 无 | 完全穿透 focus 与 silent；配 call=1 近似伪闹钟 |
| 5 | Bark timeSensitive | ❌ | ❌实测（DND 下无亮屏无声） | ✅ | — | — | 无 | 呈现被拦**但仍触发自动化** |
| 6 | 提醒 + Alert At Time | ❌ | ❌实测 | ✅ | ✅（同 #3 伴生模型） | 需人工确认 | 完成 | 到期通知；持久躺清单 |
| 7 | 日历 VALARM（订阅册） | ❌ | ❌ | ✅ | ✅（整册刷新） | ✅ | 无 | 受 Event Alerts 总开关 |
| 8 | 普通通知 / Bark active | ❌ | ❌ | ✅ | — | — | 无 | **机器门铃首选**：零打扰仍触发 |
| 9 | 静默层：清单条目（无 Alert）/ badge / 邮件 | — | — | ✅ | ✅ | — | 完成 | "等你"而非"打断你" |

**核心实测事实：通知的"呈现"与"自动化触发"完全解耦。** 触发发生在送达层，
DND/专注/通知摘要只拦呈现、拦不住触发。由此：门铃与人声分离——机器命令用
active（无光无声），给人的紧急事件才用 critical。

**一句话边界**：绝不能漏 + 必须吵醒 → 1；要打断、漏了不致命 → 2/3；计划外紧急 → 4；
要被看见、可勾掉 → 3/6；只是清单一条 → 9；机器叫机器 → 8。

## 2. severity 路由（域声明语义，网关选通道）

| severity | 落点 | 语义 |
|---|---|---|
| `critical` | 动态闹钟（既有 ES 通道） | 必须打断（还款响铃） |
| `high` | todo + **Urgent** | 必须打断 + 要完成语义 |
| `normal` | todo + Alert At Time | 到点提一声，可勾掉 |
| `low` | todo 无 Alert | 静躺清单 |
| （计划外事件·给人） | Bark critical / timeSensitive | 推通道 |
| （机器命令） | Bark active | 零打扰门铃 |

同一业务事件允许同时进闹钟通道与 todo 通道（两套 uid 空间、互不相识）。
mode（urgent/alert/silent）是**出生属性**，事后不可改——变更 = 墓碑旧条 + 新建（见 PROMPT-phone）。

## 3. 触发面与执行语义（实测）

**触发器执行模型**：单槽、串行、**忙碌即丢弃**——运行期间到达的触发全部丢失，
不排队、不合并补跑（B3/B4）。空闲时约 1.5 秒/班串行消费。
**盲窗**：重启后到首次解锁前，通知照常送达呈现，但自动化动作不落地（E3）。

**可用触发器**：时间、闹钟响/**停止**/贪睡（起床钩子：Any Alarm Is Stopped +
以 Shortcut Input 的 Label 判 `Gate-Fixed-` 前缀，勿在选择器绑死对象、勿用 Goes Off）、
睡眠日程、到达/离开地点、充电器连接/断开、电量、CarPlay、蓝牙/WiFi、NFC、App 开/关、
专注开/关、收到通知（App + Title/Message/Subtitle contains 过滤）、
收到邮件（Sender/Recipient/Subject/Account）、收到信息、声音识别。

**预留**：Alipay/微信支付通知触发 → 记账钩子；钱包交易触发。

## 4. 推/拉分工（定时器瘦身，不清零）

- **拉 = 地基（电平语义）**：已知边界时刻 + 低频心跳。计划内时刻本就是网关算的，定时器是正确工具。
- **推 = 快路径（边沿语义，不可靠）**：计划外变更 → Bark active 门铃 → 手机立即拉一轮。
  推丢了最坏退回现状，等心跳。
- **回响推送**：单槽忙丢（§3）意味着一声门铃可能撞在同步进行中而蒸发。
  网关对关键变更（god-mode、请假）**同一门铃双发**，间隔 ≥ 2 分钟（> 单轮同步最坏时长）。
  门铃幂等，多按无害。

## 5. 命令通道纪律

1. **门铃不是信件**：通知内容只做路由（关键词选指令），永不作为指令内容；
   被触发的指令一律带 key 去 GET 网关拿权威状态。泄露 Bark key 最坏 = 骚扰门铃。
2. **推送是快路径不是地基**（§4）。
3. **计划内归定时，计划外归推送**。
4. **完成语义归人**：任何自动化不得把提醒事项置为已完成。
5. **机器门铃用 active 档**：零打扰、照触发；critical 只留给必须叫醒人的事。

## 6. 实测台账（2026-07-18，iOS 26.x）

| 项 | 结论 |
|---|---|
| 删除类动作 | Remove Reminders / 删除闹钟均需人工确认，无法静默删除 |
| Set Detail | 可无人值守改 Title/Notes/Due Date/Priority/Is Completed/Is Flagged/List/Tags/URL 等；**无 Urgent 项** |
| 铃的伴生模型 | 铃随 Due Date 走：改期即改铃（R4）；**向 Due Date 喂任意非日期文本即清空日期，铃随之消灭**；移列表不消铃（E2：靠清日期消铃） |
| Add Reminder | 有 Urgent 开关与 Alert At Time；Urgent 实响穿透 DND（R2） |
| Find Reminders | URL contains 命中自定义 scheme（V1）；无 List 过滤即全库跨列表（R5）；Due Date 运算符全家桶含 is after / **is in the next**（R1）；有 Completion Date、Has (No) Alarms 过滤器 |
| Bark 三档 | active/timeSensitive 在 DND 下呈现被拦、critical 完全穿透；**三档均触发自动化**（B1） |
| 触发语义 | 通知 15/15 到达、执行 9/15：单槽串行忙丢；长任务运行期触发 30 响仅首响生效，无排队无补跑（B3/B4） |
| 执行预算 | 锁屏后台约 40~80 秒非确定被掐，掐尾部；网络往返 ~1.3s/次（国内→eu.org），本地操作 ~0.2s/次（E1） |
| 盲窗 | 重启后首次解锁前：通知送达呈现正常，自动化动作不落地（E3） |
| 通知摘要 | 由呈现/触发解耦推论：不影响门铃（B2 推论关闭） |
